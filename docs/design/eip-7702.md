# EIP-7702 Delegated Accounts

EIP-7702 lets an EOA set a persistent delegation indicator that points to
contract code. Calls to the EOA then execute the delegated code in the EOA's
own account context, preserving the address and existing assets.

Loom supports this as an adoption path, not as an upgrade authority.

## Model

1. The user signs a 7702 authorization that delegates the EOA to a reviewed
   Loom account template for the target EntryPoint.
2. The EOA sends a self-call to `initializeDelegatedAccount(...)`.
3. Initialization installs the user's validators, hooks, recovery module,
   guardian root, and config hash into the EOA's storage.
4. After initialization, the EOA follows the same Loom validation, direct
   execution, scheduling, recovery, freeze, vault, and migration rules as a
   constructor-deployed Loom account.

The initializer is callable only when `msg.sender == address(this)` and
`configVersion == 0`. A normal constructor-deployed Loom account has
`configVersion == 1`, so it cannot be reinitialized through the 7702 path.

## Security posture

- 7702 delegation is persistent. A malicious authorization can hand execution
  of the EOA to hostile code. Loom clients must never ask users to delegate to
  unreviewed code.
- Initialization is deliberately separate from the authorization tuple because
  EIP-7702 does not run initcode. A delegated EOA that has not initialized has
  no installed validator and cannot execute Loom-controlled operations.
- The delegated template address is an address-level trust decision. Users and
  clients must verify template source, runtime bytecode, compiler settings,
  EntryPoint binding, and deployment chain before signing.
- Loom does not add a registry, admin key, relayer, or recovery provider for
  7702 adoption.
- Chain-specific authorization should be preferred. Chain-agnostic 7702
  authorizations expand replay and phishing blast radius.

## Non-goals

- No automatic asset migration is needed; assets remain at the EOA address.
- No delegated-code upgrade mechanism is added. Future code changes require a
  new 7702 authorization by the EOA.
- No claim is made that 7702 is safer than a fresh Loom account. It is an
  adoption bridge for users who need address and asset continuity.
