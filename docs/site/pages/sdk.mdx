# SDK

Loom ships these early SDK packages:

- `@loom/core` for the canonical ABIs, types, hashing, manifest validation, and
  account-derivation primitives every other package builds on.
- `@loom/sdk` for the local wallet-engine client: account lifecycle intent
  builders, passkey signing, the ERC-4337 send pipeline, and app-scoped session
  helpers. The former `@loom/account` package folded into it and remains only as
  a deprecated re-export for one cycle.
- `@loom/guardian` for guardian Merkle-tree construction, inclusion proofs, and
  the guardian setup ceremony.
- `@loom/privacy` for the optional Kohaku-compatible privacy runtime boundary,
  and the canonical import point for the private-flow surface. Privacy layers on
  top of the wallet engine; a wallet that never touches a private flow pulls in
  no privacy code.

These packages are wallet-engine primitives, not a finished wallet client.
They must not hide trust assumptions, select default infrastructure, or claim
production privacy before protocol-specific review and live rehearsal.

## Account lifecycle

`@loom/account` builds explicit intents for account deployment, session grant
and revoke, recovery proposal, migration, vault withdrawal, private vault
withdrawal binding, and optional paymaster policy.

The package deliberately avoids side effects:

- no default RPC;
- no default bundler;
- no default paymaster;
- no transaction sending;
- no Loom-operated recovery coordinator.

Wallets remain responsible for explaining the intent, collecting signatures,
simulating where appropriate, and publishing through user-selected
infrastructure.

For first-run onboarding, a wallet may build a passkey-only deployment with
`recoveryStatus: 'unprotected'`. This improves onboarding but must be shown as
no guardian recovery: if the primary credential is lost before guardians are
added, access may be lost.

## Guardian tooling

`@loom/guardian` builds the guardian set off-chain. Guardians are committed
on-chain only as a Merkle root, so guardian addresses stay private until a
guardian participates (privacy-until-participation). The package computes each
guardian leaf, assembles the tree, derives the root a wallet configures on the
account, and produces the single-guardian inclusion proof that `freeze` and
recovery consume. Leaf and proof construction match the account's on-chain
`guardianLeaf` and Merkle verification, including the verifier code hash.

It also covers the setup ceremony: guardian possession challenges, a progressive
threshold setup plan, and encrypted guardian backup and onboarding evidence.

## Wallet engine client

`@loom/sdk` exposes a developer-facing client without selecting default
infrastructure:

```ts
import {
  createBundlerTransport,
  createLoomClient,
  createRpcStateTransport,
  createPasskeySigner,
} from '@loom/sdk'

// Privacy is an optional layer: a wallet that never touches a private flow
// passes no `kohaku` host and pulls in no privacy code. Add `kohaku: { host }`
// (built with `@loom/privacy`) only when a private flow is needed.
const wallet = createLoomClient({
  chainId: 1,
  account: '0x1111111111111111111111111111111111111111',
})

await wallet.sendCallsAndWait(
  {
    calls: [
      {
        target: '0x3333333333333333333333333333333333333333',
        value: 0n,
        data: '0x1234',
      },
    ],
  },
  {
    signer: createPasskeySigner({
      credentialId: 'local-credential-id',
      rpId: 'wallet.example',
      signChallenge: walletPasskeyPrompt,
    }),
    transport: createBundlerTransport({
      endpoint: 'https://bundler.example',
      entryPoint: '0x3333333333333333333333333333333333333333',
    }),
    timeoutMs: 60000,
  },
)
```

The client supports `prepareDeployAccount`, `prepareCalls`,
`prepareUserOperation`, `sendCalls`, `sendCallsAndWait`, `estimateCalls`,
`waitForUserOperationReceipt`, app-scoped `grantSession`, recovery proposal and
cancellation builders, vault withdrawal builders, private vault operation
binding, account safety state reads, typed lifecycle calldata encoders, viem-compatible call shaping,
middleware hooks, explicit bundler transport, and passkey signer boundaries.

All network, signing, paymaster, and privacy operations remain adapter-driven.
There is no default Loom RPC, bundler, paymaster, signer, or relayer.

## Recovery onboarding state

Progressive recovery lets a user start with a passkey before guardians are
configured. Wallets must still show that state clearly. `@loom/sdk` provides a
read-only account safety snapshot that uses caller-supplied infrastructure:

```ts
const state = await wallet.readSafetyState({
  stateTransport: createRpcStateTransport({
    endpoint: 'https://rpc.example',
  }),
  recoveryModule: '0x6666666666666666666666666666666666666666',
  blockTag: 'safe',
})

if (state.status === 'unprotected-recovery') {
  showWarning(state.review.summary)
}
```

The snapshot includes guardian root and threshold, validator count, freeze
status, pending recovery, and pending migration. Inconsistent guardian state or
zero validators are rejected instead of being displayed as safe. This keeps the
onboarding UX easy without hiding the cost of not configuring recovery.

## Typed calldata and viem-shaped calls

SDK users can build common Loom lifecycle calldata locally and pass it into
their own simulation, viem, bundler, or signing flow:

```ts
const data = wallet.sdk.encoders.account.revokeTokenAllowance({
  token: '0x4444444444444444444444444444444444444444',
  spender: '0x3333333333333333333333333333333333333333',
})

const calls = wallet.toViemCalls(
  wallet.prepareCalls({
    calls: [{ target: wallet.account, value: 0n, data }],
  }),
)
```

The encoder surface is deliberately local. It does not select an RPC, signer,
bundler, paymaster, relayer, or privacy provider.

## Privacy runtime

The runtime gives wallet builders a place to host Kohaku plugins behind:

- explicit provider consent;
- metadata-budget checks;
- local storage boundaries;
- no default RPC query before user consent;
- local private scan lifecycle state;
- a shielded-pool adapter wrapper;
- a Railgun profile boundary;
- future privacy-pools, stealth, and Aztec adapter surfaces.

## Provider profile

```ts
import {
  createConsentStore,
  createKohakuHost,
  createProviderProfile,
} from '@loom/privacy'

const profile = createProviderProfile({
  mode: 'user-rpc',
  chainId: 1,
  endpoint: 'https://rpc.example',
  verified: false,
  metadataBudget: {
    protocol: 'railgun',
    chainId: 1,
    items: [
      {
        surface: 'rpc',
        reveals: 'target chain and request timing',
        required: true,
        mitigation: 'user-selected endpoint',
      },
    ],
  },
})
```

## Consent before network access

```ts
const consent = createConsentStore()
const host = createKohakuHost({
  providerProfile: profile,
  consentStore: consent,
  metadataPolicy: {
    allowedSurfaces: ['rpc'],
    requireKnownMitigation: true,
  },
})

// Throws until the wallet grants consent.
await host.network.fetch(profile.endpoint)
```

## SDK package roadmap

| Package | Purpose | Status |
| --- | --- | --- |
| `@loom/account` | Account deploy, session grant/revoke, recovery proposal, migration, vault withdrawal, private vault binding, and paymaster policy builders. | Implemented boundary |
| `@loom/guardian` | Guardian Merkle-tree and inclusion-proof construction, setup ceremony, possession challenges, and encrypted backup. | Implemented boundary |
| `@loom/privacy` | Kohaku host, provider profiles, metadata budgets, local scan lifecycle, shielded-pool wrapper, and Railgun profile boundary. | Implemented boundary |
| `@loom/sdk` | Wallet engine client, app-scoped sessions, passkey signer boundary, bundler transport, gas estimation, receipt waiting, middleware, and private-vault binding. | Implemented boundary |
| Production wallet client | Mobile/web app with verified state, browser WebAuthn implementation, simulation, clear signing, live bundler qualification, and concrete privacy protocol flows. | Planned |

SDKs should improve UX without becoming account authority.
