# Security

:::warning
Loom is security-critical, pre-audit software. Passing CI is useful evidence, not
production readiness. Do not use it to secure production assets.
:::

## Current evidence

The repository has unit and integration tests, fuzz tests, stateful invariants,
gas snapshots, static checks, SDK boundary tests, selected symbolic workflows,
and website/documentation validation.

That evidence is useful for review, but the authoritative release checklist is
[Production readiness](https://github.com/emirongrr/loom/blob/main/docs/security/production-readiness.md).

## Production gates

| Gate | Status | Why it matters |
| --- | --- | --- |
| Independent audit | Missing | Immutable accounts cannot be patched after deployment. |
| Browser/device WebAuthn fixtures | Missing | Passkey compatibility must be proven across real authenticators. |
| Live bundler qualification | Missing | ERC-4337 liveness must not depend on one hosted path. |
| Privacy adapter review | Pending | ZK systems still leak metadata through providers, relayers, timing, and scanning choices. |
| L2 proof verifier audit | Pending | Cross-chain authority must be proven, not trusted through a service. |

## Never allowed

- mandatory Loom RPC;
- mandatory paymaster;
- admin upgrade key;
- global public identity registry;
- privacy provider as account authority;
- hidden recovery provider;
- permanent external veto over legitimate account operation.

When safety and convenience conflict, Loom prefers the narrow,
independently executable path and documents its limitations.
