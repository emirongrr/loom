# Getting Started

Loom is contracts-first. Clone the repository, verify the build, then add the SDK
and client layers on top.

:::info
Node.js 22 and Foundry v1.7.1 are the supported development baseline.
:::

## Five-minute quickstart

The fastest way to see a Loom account working end to end is the clean-room
proof: it packs `@loom/core` and `@loom/sdk` into tarballs, installs them into
an **empty** project, and runs the
[minimal-account example](https://github.com/emirongrr/loom/blob/main/examples/minimal-account/README.md)
against a fresh local devnet — derive the account address locally, confirm it
on chain, deploy and execute with a passkey signature over the canonical
EntryPoint hash, then send a second operation.

```sh
git clone https://github.com/emirongrr/loom.git
cd loom
npm ci
npm run e2e:clean-room
```

The consumer script imports only the public package names (the runner rejects
repository-path imports statically), so everything it does is exactly what an
external integration can do today. The packages are not yet published to npm —
the tarball path is the supported preview until the release pipeline lands —
and the whole stack remains pre-audit: suitable for evaluation, not for
production funds.

## Set up the repository

<Steps>

### Clone and install

```sh
git clone https://github.com/emirongrr/loom.git
cd loom
npm ci
```

### Run the fast verification suite

```sh
npm run verify:quick
```

`verify:quick` runs the fast local evidence suite: documentation and website
checks, SDK tests, formatting, Solidity lint, production size checks, gas
snapshot checks, contract tests, invariants, and source-policy checks.

### Build the docs site (optional)

The site is a Vocs app under `docs/site`.

```sh
npm --prefix docs/site ci
npm run site:dev
npm run site:build
```

</Steps>

## Launching a wallet on Loom

Loom draws a firm line between what the immutable engine enforces and what a
wallet client is responsible for.

| Step | Engine responsibility | Client responsibility |
| --- | --- | --- |
| Account creation | Deploy immutable `LoomAccount` with selected validators and guardian root. | Create passkeys, explain recovery, verify bytecode, and store local metadata. |
| Daily access | Validate passkey or MFA operations under policy and hook constraints. | Build clear calls, simulate risk, and route through user-selected infrastructure. |
| Permissions | Enforce bounded session authority and immediate revocation. | Translate app requests into exact permissions the user can understand. |
| Recovery | Require guardian threshold, visibility, delay, expiry, and cancellation. | Coordinate guardian ceremony without becoming a recovery custodian. |
| Privacy | Keep core usable without privacy providers. | Use Kohaku-backed adapters, local scanning, and metadata budgets. |

For the full code-vs-roadmap split, use the repository
[implementation status](https://github.com/emirongrr/loom/blob/main/docs/status.md).

## Repository checks

Run the focused suites while iterating:

```sh
npm run site:check
npm run privacy:test
npm run sdk:test
```

Run the full gate before opening a pull request:

```sh
npm run verify
```
